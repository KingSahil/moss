# Blinky — Target Matching Heuristics & Deduplication Guide

This document describes how Blinky maps plain-text target labels generated by the LLM back to physical coordinate boxes on the user's screen.

---

## 1. Step-to-Target Matching Scoring Formula

The matching engine located in [matching.py](file:///c:/projects/Jarvis/python/utils/matching.py) calculates matching scores for every visible candidate text element. The formula combines fuzzy string similarities, source confidence levels, and contextual bonuses:

$$\text{Score} = (\text{Similarity} \times 0.94) + (\text{OCR Confidence} \times 0.06) + \sum \text{Bonuses}$$

### Similarity Scores
1. **Exact Match**: If case-normalized, punctuation-stripped target text equals candidate text: `Similarity = 1.0`
2. **Substring Match**: If target is a substring of candidate text (or vice versa): `Similarity = 0.86`
3. **Fuzzy Match**: Evaluates `difflib.SequenceMatcher.ratio()`. If ratio is $< 0.65$, it is ignored.

If the LLM includes generic UI words in `target_text`, `_target_candidates()` also tries a stripped version without words like `icon`, `button`, `tab`, `menu`, `sidebar`, or `panel`. For some control-locator instructions, `_instruction_target_candidates()` can extract a better target from quoted text or phrases such as `Click the X button`.

---

## 2. Match Bonuses & Penalties

To resolve ambiguities when multiple similar terms exist on screen, the engine applies specific bonuses and penalties:

* **Exact Match Bonus (+0.30)**: Granted for case-insensitive exact string equality.
* **Input Control Preference (+0.30)**: Applied when the step is a type/search/filter action and the element is classified as an input control (e.g. `Edit`, `TextBox`).
* **Interactive Control Bonus (+0.24)**: Applied to UIA buttons, tabs, menu items, or combo boxes when the step requires interactive click coordinates.
* **Sidebar Region Context (+0.20)**: Granted if the AI instruction references words like `"sidebar"`, `"left"`, or `"panel"`, and the candidate is located in the left-hand sidebar viewport.
* **OCR Source Bonus (+0.02)**: Minor bonus for OCR-detected text containers.
* **Size/Area Bonus (Up to +0.05)**:
  $$\text{Size Bonus} = \min\left(0.05, \frac{\text{width} \times \text{height}}{10000}\right)$$
* **Blinky UI Exclusion Penalty (-0.40)**: Applied to elements matching Blinky's own UI text labels (to prevent the AI from clicking Blinky's windows instead of the target app).

### Minimum Acceptance Threshold
A candidate match is only accepted if its final score satisfies:
$$\text{Score} \ge 0.52$$
Otherwise, the engine reports no match.

### Ambiguity Diagnostics
`find_best_match_with_score()` returns diagnostics used by the locator fast path:

```json
{
  "score": 0.94,
  "text_similarity": 1.0,
  "is_exact_text": true,
  "candidate_count": 3,
  "ambiguous_candidate_count": 1,
  "source": "uia",
  "control_type": "button"
}
```

`ambiguous_candidate_count` increases when another visible candidate has the same normalized text or a score within `0.10` of the best candidate, unless both boxes overlap the same visual target. Locator requests use this to avoid confidently highlighting the wrong duplicate button or icon.

## 3. Merge & Deduplication Matrix

Before matching, [main.py](file:///c:/projects/Jarvis/python/main.py) merges UIA elements and OCR bounding boxes:

1. **Bucket Gridding**: Coordinates are divided by $8$ pixels. Items inside the same bucket are considered duplicates, where the UIA item overrides the OCR item.
2. **UIA-OCR Calibration**: If a UIA element matches an OCR element's text on the same row, UIA coordinates are overridden with the more precise OCR bounding boxes. Input controls are bypassed and keep their original full-width bounds.
3. **Input Field Expansion**: OCR items that fall inside a UIA input control's bounding box inherit the control's bounds and get marked with `control_type = "Edit"` or `"TextBox"`. This ensures typing targets highlight correctly.

---

## 4. Step Post-Processing Pipeline

After LLM inference, the output steps list undergoes a 4-stage post-processing pipeline:

### Stage 1: Target Attachment (`attach_matches`)
Scans the steps list and runs `find_best_match` to link each step's `target_text` to its physical screen coordinate match.

### Stage 2: Navigation Skip (`skip_completed_navigation_steps`)
If step 1 is a navigation action (e.g., "click Extensions panel" or "open settings") AND step 2's target is already visible on the screen, step 1 is removed and the remaining steps are renumbered.

### Stage 3: Empty Search Target Fallback (`_fill_empty_search_targets`)
If a step represents a search/type instruction but `target_text` is empty and no match was found:
- Finds the first visible item that is an input control containing a search/filter/find placeholder.
- If none is found, falls back to the first visible element containing search keywords.
- Attaches the resolved item to the step.

### Stage 4: Step Slicing (`steps[:1]`)
Programmatically restricts the return payload to a single step, minimizing latency and enforcing Blinky's single-step reactive architecture.

---

## 5. Autopilot Safety Gate

The bounded autopilot loop in `frontend/src/lib/autopilot.ts` does not trust every AI step. It only acts on the current single step after matching has already attached a visible target box.

A step is eligible when:

- `match` exists.
- The instruction contains a safe action hint: `click`, `open`, `select`, `choose`, or `go to`.
- The instruction does not contain a blocked hint: `type`, `enter`, `search`, `submit`, `install`, `enable`, `delete`, `remove`, `buy`, `purchase`, `pay`, `sign in`, or `login`.

The loop stops when:

- there is no next instruction,
- the next target is missing or unsafe,
- the same instruction/target/box repeats after a click,
- or the configured attempt limit is reached, currently 5.

The matcher remains responsible for finding the target. Autopilot is only the bounded act-and-reobserve wrapper around that existing screen understanding.
